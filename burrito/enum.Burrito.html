pub enum Burrito<T> {

The Burrito enum provides a monadic interface to an IO handle. They encapsulate all the error and state in the messy real world so that your program can exist in a state of cheesey, carne asada goodness. Or something. I don't know; it made sense for a second. Let me find a blog post.

Wrap a tortilla around some ucky stateful thing.

Burrito::new() can take any type at all, but probably its better for you to use a type that implements Read or Write.

fn and_then<F, U>(self, f: F) -> Burrito<U> where F: FnOnce(T) -> Burrito<U>

If the burrito is tasty, apply f() to its yummy innards. If its gone bad, the same burrito will be returned.

fn or_else<F>(self, f: F) -> Burrito<T> where F: FnOnce(Error) -> Burrito<T>

If the burrito is bad, create a new burrito by apply f() to the bad stuff inside. If its still good, the same burrito will be returned.

impl Burrito<File>

fn new_from_path<P: AsRef<Path>>(path: P) -> Burrito<File>

Wrapping an existing IO handle can be good, but you still have to deal with the error-prone nature of instantiating an IO handle. This will open a file handle (or not) inside of the burrito.
